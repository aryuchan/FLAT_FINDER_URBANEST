## BUGS & ISSUES TO FIX (complete list)

### 1 · `server.js` — Backend bugs

**1a. `PATCH /api/me` — `updated_at` field listed twice in the dynamic UPDATE**
The `updates` object keys are mapped to SQL `field = ?` clauses, but
`updated_at = NOW()` is hard-coded at the end of the query AND could also
appear inside `updates` if the caller sends it. Fix: remove `updated_at`
from the dynamic field list; keep only the trailing `, updated_at = NOW()`.

**1b. `POST /api/bookings` — missing `available` reset after cancellation**
When a booking is cancelled the flat's `available` column is never updated.
When ALL non-cancelled bookings for a flat are cancelled, the flat should
be set back to `available = 1`.

**1c. `GET /api/flats` — `image_public_ids` stripped but `images` JSON column
may not be parsed for the list query**
The per-flat `JSON.parse` for `amenities` / `images` runs correctly, but
if MySQL returns them as strings (which it will when using `pool.execute`)
they must always be parsed. Add a safe try-catch parse for both in the list
endpoint (it already exists in `:id` — replicate the exact same pattern).

**1d. `DELETE /api/flats/:id` — owner role check uses loose equality
but `flat.owner_id` is a `CHAR(36)` UUID that mysql2 returns as a string**
Already using `String()` coercion — confirm this is correct and add a
comment explaining why.

**1e. `POST /api/flats` — `available_from` empty string stored as `''`
instead of `NULL`**
`available_from || null` evaluates to `null` only when the string is
falsy, but `''` (empty string) is falsy, so this actually works — however,
the comment should clarify this; add one. Also guard `deposit`, `floor`,
`total_floors`, `area_sqft` against empty strings (`'' || null`).

**1f. Global error handler must flush headers before sending JSON if the
response was already started** — add `if (res.headersSent) return;` guard.

**1g. CORS `origin === 'null'`** — This string `'null'` appears when a file
is opened as `file://` in browsers. Do NOT allow `file://` in production.
Add a guard: only allow `'null'` origin when `!IS_PROD`.

**1h. Missing `PATCH /api/me` field: `updated_at` must not be in the column
list** (same as 1a — document explicitly).

**1i. `GET /api/listings` for an owner returns `flat_title` from `f.title`
but the frontend references `l.flat_title`** — verify the SQL alias matches
exactly what `ff-owner.js` and `ff-admin.js` expect (`flat_title`,
`flat_id`, `owner_name`, `reviewer_name`). Fix any mismatch.
Current SQL: `f.title AS flat_title` ✓ — but also add `f.id AS flat_id`
since the owner dashboard uses `l.flat_id` for the delete button's
`data-flat-id`. The listings table already has `flat_id` as a column, but
the SELECT must explicitly expose it (it is exposed via `l.*` — confirm).

**1j. `POST /api/flats` — race condition on fetching the newly inserted flat**
`SELECT * FROM flats WHERE owner_id = ? ORDER BY created_at DESC LIMIT 1`
is non-deterministic under concurrent inserts. Fix: use `INSERT ... ; SELECT LAST_INSERT_ID()` — but because the PK is a UUID generated by MySQL's
`DEFAULT (UUID())`, use the pattern:
```sql
SET @new_id = UUID();
INSERT INTO flats (id, ...) VALUES (@new_id, ...);
SELECT * FROM flats WHERE id = @new_id;
```
Or simpler: wrap in a transaction and return the `id` directly from
`result.insertId` — but since UUID is generated by MySQL, the safest fix
is to generate the UUID in Node (`crypto.randomUUID()`), pass it
explicitly as the first column, then fetch by that known id.

**1k. Missing API endpoint: `GET /api/ping` should also verify DB, not just
`SELECT 1`** — already done, confirm it returns `{ success, message }` JSON
(it does — note confirmed).

### 3 · `ff-core.js` — Frontend bugs

**3a. `render()` calls `bindEvents()` which is defined in `app.js` and
loaded after `ff-core.js`** — at the time `ff-core.js` is parsed,
`bindEvents` does not exist. The `typeof bindEvents === 'function'` guard
is correct, but `render()` is also called *before* `app.js` executes
during module load — confirm this is safe (it is, because `render` is only
called after `app.js` loads). Add an inline comment.

**3b. `renderNavBar()` — nav links use `href="#route"` AND `data-route="route"`**
but the event delegation in `app.js` only intercepts `data-route` clicks
and calls `window.location.hash = "#" + route`. Meanwhile `href` causes a
real hash-change event. This means two navigations fire on every nav-link
click. Fix: remove `href` from nav links (set `href="javascript:void(0)"`
or `href="#"`) so only `data-route` drives navigation.

**3c. `showModal()` — the outer `#app-modal` div has `aria-hidden="true"`
as a static attribute in the HTML but is never toggled when the modal
opens/closes.** Fix: toggle `aria-hidden` on `#app-modal` in `showModal`
and `closeModal`.

### 8 · `app.js` — Bugs

**8a. `bindEvents()` — it clones `app-root` to remove old listeners, then
re-binds. But `Auth.bindEvents`, `Tenant.bindEvents`, etc. are called
with the cloned `root`. Meanwhile `render()` in `ff-core.js` sets
`root.innerHTML = html` on the ORIGINAL `app-root` element (found by
`getElementById`). After `cloneNode`, the cloned element is the one in
the DOM (it replaced the original via `replaceWith`), so `getElementById`
will now correctly find the clone.** Confirm this works — yes it does
because `replaceWith` puts the clone in the DOM. ✓

**8b. `navigate()` — the navigation lock `_navigating` is released in
`finally`. But if the user clicks a nav link rapidly while `_navigating`
is true, the click is silently swallowed.** This is acceptable UX — add
a comment explaining the design choice.

**8c. `defaultRoute()` is defined in `app.js` but referenced inside
`ff-auth.js` which loads before `app.js`.** This creates a temporal dead
zone during module evaluation. However, `defaultRoute` is only called
at runtime (inside `authForm.addEventListener('submit', ...)`) — not at
parse time — so by the time the event fires, `app.js` has fully executed
and `defaultRoute` is in scope. **This is safe but fragile.** Fix: move
`defaultRoute` to `ff-core.js` so it's available to all modules from the
start.

**8d. `handleLogout` is defined in `app.js` but called from `ff-core.js`'s
`renderNavBar()` — same temporal-dead-zone risk as 8c. Same fix: move
`handleLogout` to `ff-core.js`.**
